# 大模型微调

------

## 1. 先建立总框架：微调到底在解决什么问题？

> 预训练决定模型的通用语言能力，微调决定模型在具体任务、具体风格、具体业务约束下怎么表现。  
> 所以微调本质上不是“让模型重新学会世界知识”，而是让模型在特定目标上更稳、更像你想要的样子。

面试里最好先把下面四个概念分清：

| 方案 | 解决的核心问题 | 典型场景 | 代价 |
| --- | --- | --- | --- |
| Prompt | 临时引导模型行为 | 快速试验、低成本上线 | 最低 |
| RAG | 接入外部和实时知识 | 客服、医疗、法律、企业知识库 | 中 |
| SFT/PEFT | 固化行为、格式、风格、任务能力 | 分类、抽取、结构化输出、工具调用 | 中到高 |
| 偏好对齐 | 让回答更符合人类偏好和可验证目标 | Chat 助手、推理、奖励驱动任务 | 高 |

一个好回答通常会补一句：

> 实际工程里我会优先按 `Prompt -> RAG -> 微调 -> 偏好对齐` 逐步加复杂度，而不是一上来就训练。

------

## 2. 面试第一问：什么时候该微调，什么时候不该微调？

### 适合微调的场景

- 需要稳定输出固定格式，比如 JSON、SQL、函数调用参数
- 需要固定语气、品牌风格、角色设定
- 需要做分类、抽取、打分、排序这类任务适配
- 需要把工具调用协议、流程模式“写进模型习惯”
- Prompt 很长、很脆弱，且长期重复调用

### 不适合先靠微调解决的场景

- 业务知识更新很快
- 需要可溯源、可引用原文
- 数据量很小，且任务本质是缺知识不是缺行为
- 问题更像“系统设计问题”而不是“模型能力问题”

### 一句话回答模板

> 如果问题本质是“模型不知道最新知识”，我优先上 RAG；如果问题本质是“模型知道，但老是不按我想要的方式做”，我才优先考虑微调。

------

## 3. 微调家族总览

### 3.1 继续预训练 / Continued Pretraining

> 继续预训练更适合“补领域知识”，比如法律、医疗、金融语料的长期吸收。  
> 它训练的是语言建模目标，本质上是在补模型底座，不是直接做指令对齐。

### 3.2 SFT

> SFT 是监督微调，用高质量 `instruction -> answer` 数据让模型学会更好地遵循指令。

### 3.3 PEFT

> PEFT 是参数高效微调，核心目标是在不改大部分底座参数的情况下，用很小的代价完成任务适配。LoRA/QLoRA 是最常见方法。

### 3.4 偏好对齐

> 偏好对齐解决的是“多个答案都能说得通，但哪个更好”的问题，典型方法是 RLHF、DPO、ORPO、SimPO、GRPO。

### 3.5 一个常见误区

> 很多人把“补知识”和“调行为”混在一起讲。严格来说，领域知识吸收更偏继续预训练；指令遵循和格式风格更偏 SFT；偏好和可验证奖励更偏对齐训练。

------

## 4. SFT：面试最常问的基础

### 4.1 什么是 SFT？

> SFT 就是监督微调。给模型输入指令和参考答案，让模型继续学习“在这种输入下应该怎么输出”。

### 4.2 SFT 的目标函数是什么？

> 本质上还是语言模型的 next-token prediction，只不过训练样本换成了更贴近任务的数据。常见实现仍然是交叉熵损失。

### 4.3 一个标准 SFT 流程怎么讲？

1. 选基础模型
2. 构造高质量指令数据
3. 处理模板和 chat format
4. 配置训练参数
5. 训练并观察 loss/验证集
6. 离线评测
7. 线上灰度和回滚

### 4.4 面试回答模板

> 如果让我从零做一版 SFT，我会先明确任务目标，再把数据整理成统一的指令格式，保证风格一致和标签可靠。训练时从小学习率开始，配合 warmup、梯度累积、验证集早停。最后不会只看 loss，而是结合真实业务评测集做自动评测和人工抽检。

------

## 5. SFT 的数据设计：真正拉开差距的地方

### 5.1 常见数据格式

最常见的思路是：

```json
{
  "instruction": "请总结下面文章",
  "input": "...",
  "output": "..."
}
```

或者直接使用 chat 格式：

```json
[
  {"role": "system", "content": "..."},
  {"role": "user", "content": "..."},
  {"role": "assistant", "content": "..."}
]
```

### 5.2 数据质量比数据量更重要

优先保证这几件事：

- 标签正确
- 风格一致
- 指令边界清晰
- 噪声少
- 覆盖主场景和长尾场景

### 5.3 训练数据最容易踩的坑

- 同一任务混入多种冲突风格
- 输入和输出边界不清
- 标签抄原文但没经过质量校验
- 样本高度重复，导致模型背答案
- 只做头部样本，不做难例和反例

### 5.4 数据构造的工程经验

我一般会补这几个点，面试官很爱听：

- 先做小规模高质量种子集
- 用真实线上 badcase 反补训练集
- 把格式错误、工具调用错误、拒答错误单独成类
- 留独立验证集，不能训练评测不分家

------

## 6. SFT 训练细节：高频参数怎么讲？

### 学习率

> 大模型微调学习率通常比预训练低很多。经验上 SFT 常从 `1e-5 ~ 2e-5` 量级起步，小模型和 Adapter 训练可以略高，但不能死背固定数值，要看 batch、数据量和训练目标。

### Batch Size 和梯度累积

> 真正重要的是有效 batch size。显存不够时，用梯度累积去模拟更大的 batch。

### 最大长度

> 最大长度不要盲目拉满，要看训练数据长度分布。过长会浪费显存，过短会截断关键信息。

### Warmup 和学习率衰减

> Warmup 能降低训练初期不稳定，常和 cosine 或 linear scheduler 搭配。

### Epoch

> SFT 通常 1 到 3 个 epoch 就够了，再多很容易过拟合。

### 早停

> 训练损失持续下降但验证集指标不涨，甚至下降，就该考虑早停。

------

## 7. 全参数微调 vs PEFT：怎么选？

| 维度 | 全参数微调 | PEFT / LoRA |
| --- | --- | --- |
| 显存成本 | 高 | 低 |
| 训练速度 | 慢 | 快 |
| 多任务管理 | 差 | 好 |
| 极限性能 | 理论上更强 | 通常够用 |
| 灾难性遗忘风险 | 更高 | 更低 |

### 标准回答

> 绝大多数业务场景下，我会先选 LoRA/QLoRA 做实验和上线，因为它便宜、快、适合多任务版本管理。只有在数据非常多、算力非常足、对极限效果追求很高时，才考虑全参数微调。

## 8. LoRA：面试一定会问

### 8.1 LoRA 的核心思想

> LoRA 假设模型在适配新任务时，不需要更新整个大矩阵，只需要学习一个低秩增量。  
> 所以它不是直接训练完整的 `ΔW`，而是把它拆成两个更小的矩阵 `B @ A`。

原始层：

```python
h = W x
```

LoRA 后：

```python
h = W x + (alpha / r) * B A x
```

### 8.2 为什么这样做有效？

- 可训练参数更少
- 显存更低
- 训练更快
- 更适合多任务挂 Adapter
- 对底座破坏小，灾难性遗忘更轻

### 8.3 LoRA 常加在哪些层？

> 最常见是 attention 里的 `q_proj`、`v_proj`，有时也会加 `k_proj`、`o_proj` 和 MLP 层。  
> 工程上通常先从 `q_proj + v_proj` 起步，因为这是性价比最高的 baseline。

### 8.4 Rank 怎么选？

> `r` 太小，表达能力不够；太大，显存和过拟合风险上来。  
> 工程上常从 `r=8` 或 `r=16` 起步，小任务够用；更复杂任务再上调。

### 8.5 高频面试回答

> LoRA 的本质是用低秩更新逼近全量更新，以更小的参数量完成任务适配。它最大的工程价值不是数学优雅，而是让大模型微调从“大厂专属”变成“团队可落地”。

------

## 9. QLoRA：为什么单卡也能调大模型？

### 9.1 核心思路

> QLoRA 不是把 LoRA 量化，而是把基础模型权重量化到 4-bit 存储，同时保留 LoRA Adapter 可训练，从而大幅降低显存。

### 9.2 为什么效果还行？

> 因为底座虽然低比特存储，但训练时关键计算仍会在更高精度里做；真正更新的还是小规模 LoRA 参数。

### 9.3 QLoRA 的优点

- 显存占用大幅降低
- 单卡训练变得可行
- 效果通常接近全精度 LoRA

### 9.4 QLoRA 的代价

- 训练速度不一定总是更快
- 某些任务对量化更敏感
- 调参和环境兼容性更复杂

### 9.5 面试版回答

> QLoRA 的价值在于把“是否能训练”这个门槛先打下来。对很多团队来说，QLoRA 不是极限最优，但往往是性价比最优。

------

## 10. 继续预训练和 SFT 的区别

这个问题现在越来越常见。

### 继续预训练更适合

- 领域术语多
- 领域知识密度高
- 需要长期吸收行业语料
- 想增强底层知识分布适配

### SFT 更适合

- 指令遵循
- 输出格式
- 风格和角色
- 分类、抽取、问答等任务适配

### 一句话区分

> Continued Pretraining 更像“让模型先多读专业书”，SFT 更像“教模型按要求答题”。

------

## 11. 为什么只做 SFT 还不够？

> 因为 SFT 学到的是“像标准答案”，不是“更符合人类偏好”。  
> 很多问题没有唯一标准答案，但存在明显更优的回答，比如更安全、更简洁、更有帮助、更符合规则。

所以 SFT 常见局限有：

- 会模仿，不会比较优劣
- 对长尾偏好覆盖差
- 很难显式优化“更安全”或“更像人”
- 容易学成平均答案

------

## 12. RLHF：经典但复杂

### 三阶段

1. 先做 SFT，得到基础助手
2. 用偏好数据训练奖励模型
3. 用 PPO 等强化学习方法优化策略模型

### RLHF 的优点

- 理论完整
- 能直接引入奖励信号
- 适合复杂偏好目标

### RLHF 的难点

- 流程长
- 训练不稳定
- 超参数敏感
- 容易出现 reward hacking

### 面试回答

> RLHF 的核心不是“强化学习很高级”，而是它把“人类更喜欢哪个回答”转成了训练信号。但工程上它很重，所以很多团队更愿意从 DPO 这类简化方法入手。

------

## 13. DPO：现在更常见的偏好优化入门方案

### 13.1 DPO 在做什么？

> DPO 直接用偏好对数据训练模型，让模型提升对优选回答的概率，压低对劣选回答的概率，不再显式训练奖励模型。

### 13.2 为什么 DPO 受欢迎？

- 流程比 RLHF 短
- 训练更稳定
- 更像普通监督训练
- 更适合中小团队快速落地

### 13.3 什么数据能喂给 DPO？

常见格式是：

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "..."
}
```

### 13.4 面试版回答

> 如果我有高质量偏好对数据，且目标是把一个 SFT 模型进一步对齐成人类偏好，DPO 通常是工程上更稳、更轻的首选。

补一句更像做过工程的话：

> 按 Hugging Face 当前的 TRL 文档，DPO 训练已经支持标准 preference dataset，也支持 conversational dataset；如果是工具调用或多轮对话，对数据格式和模板一致性要特别小心。  
> 来源：<https://huggingface.co/docs/trl/en/dpo_trainer>

------

## 14. ORPO、SimPO、GRPO：近两年常见加分点

这部分不一定每场都问，但答出来会显得你不是只停留在 LoRA 和 DPO。

### ORPO

> ORPO 可以理解为把监督学习和偏好优化揉得更紧，不再依赖单独参考模型，工程链路更短。

### SimPO

> SimPO 进一步走“reference-free”路线，直接用序列平均 log probability 做隐式奖励，目标是让偏好优化更简单、更省算力。

### GRPO

> GRPO 更常出现在“可验证奖励”的推理训练里，比如数学、代码、规则明确的任务。它通过组内相对优势来做优化，近一年在 reasoning post-training 讨论度很高。

### 面试里怎么说更稳

> 现在偏好优化不只有 DPO 一条路。工程上常见还是 DPO；如果做更轻量的 reference-free 优化，可以关注 ORPO、SimPO；如果是可验证奖励的推理任务，GRPO 也是近一年的热点方向。

注意：

> 这里我说的是“趋势和适用场景”，不是说这些方法已经全面替代 DPO。通用对话场景下，DPO 仍然是非常主流的落地选择。

再补一个“工具链视角”的点：

> 从当前 Hugging Face TRL 文档看，ORPO 和 GRPO 都已经有独立 trainer，SimPO 也已经以 loss 形式进入官方训练工具链。这说明这些方法不只是论文概念，已经逐步进入可复现、可工程化阶段。  
> 来源：<https://huggingface.co/docs/trl/orpo_trainer> 、<https://huggingface.co/docs/trl/en/grpo_trainer> 、<https://huggingface.co/docs/trl/main/en/trainer>

## 15. RLAIF 是什么？

> RLAIF 是用 AI feedback 替代人工偏好标注。也就是让强模型或规则系统帮你给回答打偏好信号。

优点：

- 省人工标注
- 扩数据更快

风险：

- 容易把教师模型偏见放大
- 反馈质量不稳定
- 需要抽检和混入人工监督

------

## 16. 微调中的评估怎么做？

### 自动评估

- 分类任务：Accuracy、F1、AUC、LogLoss
- 生成任务：ROUGE、BLEU、Exact Match
- 指令遵循：格式命中率、JSON 合法率、工具调用成功率
- 偏好对齐：Win Rate、pairwise preference accuracy

### 人工评估

我一般看四个维度：

- 正确性
- 完整性
- 风格一致性
- 安全性

### 真正重要的一点

> 不要只看训练 loss。训练 loss 下降只代表模型更会拟合训练集，不代表业务效果一定更好。

## 17. 微调最常见的工程坑

### 17.1 灾难性遗忘

> 模型过度适配新任务，损失了原有通用能力。  
> 应对方法包括：PEFT、混入通用数据、控制学习率和训练步数。

### 17.2 过拟合

典型信号：

- 训练集效果越来越好
- 验证集效果停滞或变差

应对方法：

- 降学习率
- 降 rank
- 减 epoch
- 提高数据多样性
- 做早停

### 17.3 数据泄漏

> 训练集和验证集样本高度相似，导致离线分数虚高。

### 17.4 模板不一致

> 训练时一种 chat template，推理时另一种，效果会明显掉。

### 17.5 只会看 benchmark，不看真实业务集

> 真正上线前必须做贴近业务的 eval set，不然很容易线下高分、线上翻车。

------

## 18. 微调与 RAG 可以怎么组合？

这是应用开发面试特别爱问的题。

### 组合思路

- RAG 负责补知识
- SFT 负责稳格式和风格
- DPO/偏好优化负责提高回答偏好
- Agent/Workflow 负责多步流程

### 一句话回答

> RAG 解决“知道什么”，微调解决“怎么说、怎么做”，两者不是互斥关系，而是经常组合使用。

------

## 19. 一个可直接背的项目回答模板

> 我们一开始先用 Prompt + RAG 跑 baseline，解决知识接入问题。上线后发现模型虽然能查到信息，但输出格式不稳定、工具调用经常错，于是我们做了基于 LoRA 的 SFT，把结构化输出和流程约束固化下来。之后如果发现多个回答都基本正确，但用户更偏好某种风格或更安全的答案，再继续补 DPO 这类偏好优化。整个过程不是一次性上最复杂方案，而是分阶段增加复杂度。

## 20. 高频问答速记

### Q1：全参数微调和 LoRA 你怎么选？

> 默认先 LoRA/QLoRA，便宜、快、可迭代；极限性能和大数据场景再考虑全参数。

### Q2：LoRA 为什么是低秩？

> 因为它假设任务适配不需要完整更新整个大矩阵，低秩增量已经能表达主要变化。

### Q3：QLoRA 为什么省显存？

> 因为底座权重 4-bit 存储，真正训练的是小规模 Adapter。

### Q4：什么时候要继续预训练？

> 当问题是领域知识不足，而不是行为不对时。

### Q5：为什么只做 SFT 不够？

> SFT 学的是模仿标准答案，不会直接学“哪个答案更优”。

### Q6：DPO 相比 RLHF 的优势是什么？

> 流程更短，训练更稳，实现成本更低。

### Q7：GRPO 更适合什么？

> 更适合有可验证奖励的推理任务，比如数学、代码和规则明确的问题。

### Q8：微调效果怎么评估？

> 自动指标 + 人工评估 + 真实业务集，三者都要有。

------

## 21. 这份文档最值得背的 10 句话

- 微调不是补一切，先判断问题是知识问题还是行为问题。
- Continued Pretraining 更偏补知识，SFT 更偏调行为。
- SFT 的核心是高质量数据，不是盲目堆数据量。
- LoRA 的价值首先是工程效率，其次才是数学形式。
- QLoRA 的核心贡献是把门槛打下来。
- SFT 解决“会做”，偏好优化解决“做得更好”。
- DPO 是最常见的轻量偏好优化方案。
- ORPO、SimPO、GRPO 是近两年值得知道的进阶方法。
- 训练 loss 下降不等于业务效果变好。
- RAG 和微调不是二选一，通常是组合拳。
