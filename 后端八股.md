# 后端八股

------

## 1. 这份文档怎么定位？

这份后端八股不是只讲大模型，也不是只讲传统后端，而是按“两层结构”准备：

- 第一层：传统后端高频基础
- 第二层：大模型应用后端的特有问题

如果面试是普通后端岗，你至少要把前 15 节讲顺；如果是大模型应用开发岗，最好把后半部分也吃透。

------

## 2. 面试官为什么会在大模型岗位里问传统后端？

> 因为模型能力再强，也要落在服务、接口、存储、缓存、任务调度和监控上。  
> 真正上线以后，很多问题都不是“模型不会答”，而是“接口慢、缓存错、权限漏、任务重复、下游超时、数据库扛不住”。

所以我对后端的理解是：

> 传统后端解决“系统怎么稳定运行”，大模型后端是在这个基础上，再解决“模型怎么稳定可控地运行”。

------

## 3. 一个完整后端系统通常包括哪些层？

我一般按 6 层来讲：

1. 接入层：Nginx、网关、鉴权、限流、负载均衡
2. 应用层：业务服务、工作流、接口编排
3. 数据层：MySQL/PostgreSQL、Redis、对象存储、向量库
4. 异步层：MQ、任务队列、定时任务
5. 基础设施层：容器、服务发现、配置中心
6. 可观测层：日志、指标、Tracing、告警

传统后端和 LLM 后端的骨架其实一样，只是 LLM 系统会在应用层和能力层多出模型调用、RAG、Agent、上下文治理。

------

## 4. 从用户请求进入系统，到底经过了什么？

这是面试里很实用的回答框架：

1. 客户端发起 HTTP 请求
2. Nginx / 网关接入并做路由、鉴权、限流
3. 应用服务接收请求并做参数校验
4. 查询缓存，未命中再查数据库或下游服务
5. 如果是异步任务，则投递到消息队列
6. 结果写库或返回前端
7. 全链路记录日志、指标和 trace

如果是 LLM 场景，就在第 4 步和第 5 步之间再插入：

- 检索
- 重排
- Prompt 构造
- 模型调用
- 工具调用

------

## 5. HTTP 和 HTTPS 的区别？

> HTTP 是明文传输，HTTPS = HTTP + TLS，加了加密、身份认证和完整性保护。

面试里建议补三个点：

- HTTPS 能防窃听
- HTTPS 能防中间人篡改
- HTTPS 握手更复杂，但现在是标配

如果继续追问，可以说：

> 现代服务里 HTTPS 已经不是“要不要用”，而是“证书、终止位置、代理链路怎么配”的问题。

------

## 6. TCP 和 UDP 的区别？为什么后端常用 TCP？

> TCP 面向连接、可靠传输、有序；UDP 无连接、快，但不保证可靠和顺序。

绝大多数 Web 后端基于 TCP，因为接口服务更关注可靠性和正确性。UDP 更常见于音视频、实时通信、DNS 这类场景。

------

## 7. TCP 三次握手和四次挥手怎么理解？

### 三次握手

目的不是背流程，而是理解两件事：

- 双方都确认对方可收可发
- 初始化连接状态和序列号

### 四次挥手

因为 TCP 是全双工，双方关闭发送通道通常要分开确认，所以常见是四次。

面试里如果你不想背细节，可以一句话概括：

> 握手是为了建立可靠连接，挥手是为了双方有序地结束连接。

------

## 8. Cookie、Session、JWT 分别是什么？怎么选？

### Cookie

> Cookie 是浏览器存储在本地的小数据，后续请求会自动带上。

### Session

> Session 是服务端保存的会话状态，客户端通常只带 Session ID。

### JWT

> JWT 是自包含的令牌，服务端不一定需要查 session 存储就能校验身份。

### 怎么选

- 传统 Web 登录态：Session 很常见
- 分布式 / API 网关场景：JWT 更常见
- 高安全要求：别只看“无状态好不好”，还要看过期、撤销、刷新和密钥管理

------

## 9. REST、RPC、SSE、WebSocket 怎么选？

### REST

> 面向资源、通用性强，适合大多数普通业务接口。

### RPC

> 更强调服务间调用性能和接口契约，微服务内部常见。

### SSE

> 基于 HTTP 的服务端单向流式推送，很适合大模型 token streaming。

### WebSocket

> 双向长连接，适合实时互动、高频双向通信。

### 一句话记忆

> 普通接口优先 REST，内部服务高频调用可考虑 RPC，LLM 流式输出常用 SSE，强实时双向交互再用 WebSocket。

------

## 10. 为什么很多大模型聊天接口优先用 SSE？

MDN 对 SSE 的描述很明确：它是基于 HTTP 的单向事件流，适合服务端持续推送数据；`EventSource` 也是浏览器原生支持的单向事件接口。基于这个特性，聊天输出这种“服务端不断吐 token”的场景和 SSE 很匹配。  
来源：<https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events> 、<https://developer.mozilla.org/docs/Web/API/EventSource>

工程上它比 WebSocket 更轻，但也要注意：

- 反向代理超时
- 客户端断连
- 流结束标记
- 代理缓冲
- 心跳保活

------

## 11. GET 和 POST 有什么区别？

> GET 更偏获取资源，POST 更偏提交数据和产生副作用。

面试里常见延伸点：

- GET 通常应幂等，POST 不一定
- GET 参数常在 URL，POST 参数常在 body
- GET 更容易被缓存，POST 默认不强调缓存
- 这不是“GET 不能传 body、POST 一定不幂等”这种绝对规则，而是语义和工程约定问题

------

## 12. 常见 HTTP 状态码怎么答？

高频的够用了：

- `200`：成功
- `201`：创建成功
- `301/302`：重定向
- `400`：请求参数错误
- `401`：未认证
- `403`：已认证但无权限
- `404`：资源不存在
- `429`：请求过多，被限流
- `500`：服务端内部错误
- `502/503/504`：网关错误、服务不可用、网关超时

如果是大模型接口，`429`、`503`、`504` 会特别常见，因为模型供应商限速和下游超时都很普遍。

------

## 13. 什么是幂等？为什么重要？

> 幂等的意思是同一个请求执行一次和执行多次，结果应该一致，至少业务副作用不能重复。

典型场景：

- 订单重复提交
- 支付回调重复到达
- MQ 重复消费
- 用户多次点击“提交”
- LLM 异步报告重复生成

实现方式常见有：

- 幂等键
- 去重表
- 唯一索引
- 状态机控制

------

## 14. 什么是限流？常见算法有哪些？

> 限流是保护系统稳定性和成本边界的手段。

常见算法：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

面试里常用回答：

> 令牌桶更适合允许一定突发流量，漏桶更适合严格整流。

在 LLM 场景里，限流不止是 QPS，还包括：

- 并发数
- token 数
- 租户级预算

------

## 15. 负载均衡是什么？常见策略有哪些？

> 负载均衡就是把请求合理分发到多台机器，提升吞吐、可用性和扩展性。

常见策略：

- 轮询
- 加权轮询
- 最少连接
- 一致性哈希
- 按地理位置或机房路由

如果继续问到 Nginx 和服务注册中心：

> Nginx 更常做七层反向代理，注册中心更偏服务发现，二者经常一起用。

------

## 16. Redis 常见用途有哪些？

不要只答“缓存”，更完整的说法是：

- 缓存
- 分布式锁
- 限流计数
- 排行榜
- 会话状态
- 延迟队列的简化实现
- 大模型场景下的热 query / embedding / Prompt 缓存

------

## 17. 缓存穿透、击穿、雪崩分别是什么？

### 缓存穿透

> 查一个根本不存在的数据，请求每次都打到数据库。

常见解决方案：

- 布隆过滤器
- 空值缓存

### 缓存击穿

> 一个热点 key 失效瞬间，大量请求同时打到数据库。

常见解决方案：

- 热点 key 永不过期或逻辑过期
- 互斥锁 / singleflight

### 缓存雪崩

> 大量 key 同时失效，数据库瞬间被打爆。

常见解决方案：

- TTL 打散
- 多级缓存
- 降级兜底

------

## 18. MySQL 索引为什么能加速？B+ 树为什么常用？

> 因为索引本质上是帮你减少扫描的数据量。B+ 树适合磁盘和范围查询，非叶子节点只存索引键，叶子节点有序，范围扫描效率高。

面试里常见延伸点：

- 聚簇索引和二级索引
- 回表
- 覆盖索引
- 最左前缀匹配

### 一句话回答

> MySQL 常用 B+ 树，是因为它兼顾了单点查找、范围查询和磁盘 I/O 效率。

------

## 19. 什么是回表？什么是覆盖索引？

### 回表

> 通过二级索引找到主键后，再回到主键索引或数据页取完整记录。

### 覆盖索引

> 查询需要的字段都在索引里，不需要回表。

工程上常说“尽量让高频查询命中覆盖索引”，本质上就是减少随机 I/O。

------

## 20. 最左前缀原则怎么理解？

> 联合索引从最左列开始生效。查询条件如果跳过最左列，索引效果通常会大打折扣。

面试里别死背定义，直接结合例子说更好：

> 如果索引是 `(a, b, c)`，那么查 `a`、`a+b`、`a+b+c` 通常都能较好利用索引，但直接查 `b+c` 往往不行。

------

## 21. 事务的 ACID 是什么？

- Atomicity：原子性
- Consistency：一致性
- Isolation：隔离性
- Durability：持久性

面试里如果只会背缩写不够，最好加一句：

> 事务的目标是在并发和异常下，保证一组操作要么一起成功，要么一起失败，并且最终状态可恢复、可解释。

------

## 22. 事务隔离级别有哪些？幻读、不可重复读是什么？

常见隔离级别：

- Read Uncommitted
- Read Committed
- Repeatable Read
- Serializable

### 不可重复读

> 同一事务里两次读同一行数据，结果不一样，通常因为别的事务更新并提交了。

### 幻读

> 同一事务里两次按条件查询，第二次多出或少了满足条件的行。

MySQL InnoDB 默认是 Repeatable Read，并结合 MVCC 和间隙锁处理并发问题。

------

## 23. 什么是 MVCC？

> MVCC 是多版本并发控制，核心思想是让读不阻塞写、写不阻塞读太多，通过数据版本和 undo log 来实现一致性读。

面试里这样答通常够用：

> MVCC 主要是提升并发读性能，是 InnoDB 能在较高并发下保持可用性的关键机制之一。

------

## 24. 乐观锁和悲观锁怎么理解？

### 悲观锁

> 先锁住资源，再操作，默认认为冲突很常见。

### 乐观锁

> 不先加重锁，而是在更新时校验版本号或条件，默认认为冲突不常见。

### 怎么选

- 写冲突高、数据强一致要求高：更偏悲观锁
- 读多写少、并发高：更偏乐观锁

常见例子：

- 数据库 `select ... for update` 偏悲观锁
- `version` 字段更新偏乐观锁

------

## 25. 主从复制、读写分离、分库分表分别是什么？

### 主从复制

> 主库写，从库同步，主要解决读压力和高可用问题。

### 读写分离

> 写请求走主库，读请求走从库。

### 分库分表

> 当单库单表数据量、并发量、索引体积太大时，把数据拆散。

面试里可以这样总结：

> 主从复制和读写分离主要是扩读能力，分库分表主要是扩容量和扩写能力，但会带来事务、聚合查询、分页和运维复杂度。

------

## 26. 为什么需要消息队列？

> 因为不是所有任务都适合同步做。消息队列的核心价值是解耦、削峰、异步、重试。

常见场景：

- 下单后发通知
- 日志异步处理
- 文档解析和建库
- 批量 embedding
- 报告生成
- Agent 长链路任务

别忘了补风险点：

- 消息重复
- 顺序消费
- 消息积压
- 死信队列
- 幂等消费

------

## 27. 什么是 CAP 和 BASE？

### CAP

> 分布式系统里，一致性、可用性、分区容错性不能同时极致满足，网络分区存在时通常要在一致性和可用性之间做权衡。

### BASE

> Basically Available、Soft state、Eventually consistent。它强调的是工程上的最终一致，而不是强一致。

面试里可以这样说：

> 业务系统一般不是追求“绝对强一致”，而是按场景判断哪些地方必须强一致，哪些地方可以最终一致。

------

## 28. 什么是分布式锁？Redis 锁有什么风险？

> 分布式锁是为了让分布式环境下多个实例对共享资源的操作互斥。

Redis 锁常见风险：

- 锁过期导致并发进入
- 误删别人的锁
- 主从切换时锁丢失

所以面试里最好补一句：

> 分布式锁不是万能解，很多场景更优先考虑幂等、状态机和唯一约束。

------

## 29. 什么是服务雪崩、熔断、降级？

### 服务雪崩

> 一个下游挂掉，层层放大，拖垮整个系统。

### 熔断

> 下游连续失败时，暂时不再继续请求，防止故障扩散。

### 降级

> 在部分能力不可用时，退回更简单但还能工作的方案。

### 在大模型系统里的典型降级

- 主模型失败切备模型
- Reranker 超时则只走召回
- 长任务转异步
- 工具失败退回模板回复

------

## 30. 什么是可观测性？和“打日志”有什么区别？

> 可观测性不只是日志，而是日志、指标、Tracing 三件事一起做。

### 日志

看具体事件和上下文。

### 指标

看聚合趋势，比如 QPS、错误率、P95。

### Trace

看一次请求在各个服务之间到底走了什么链路。

面试里建议这样答：

> 日志适合看个例，指标适合看趋势，Trace 适合看链路，三者缺一不可。

------

## 31. 传统后端和大模型后端最大的区别是什么？

> 传统后端主要处理的是“数据库 + 业务逻辑 + 服务稳定性”；大模型后端在此基础上，还要处理模型调用成本、长链路延迟、上下文构造、RAG 检索质量、Agent 工具调用和不可预测输出。

也就是说：

> 大模型后端不是替代传统后端，而是在传统后端上再加一层“模型工程复杂度”。

------

## 32. 一个典型的大模型应用后端架构是什么样的？

我一般会拆成 6 层：

1. 接入层：网关、鉴权、限流、路由
2. 应用层：Prompt 编排、工作流、Agent 状态机
3. 能力层：模型调用、RAG、工具调用、重排、缓存
4. 数据层：MySQL / PostgreSQL、Redis、对象存储、向量库
5. 异步层：消息队列、离线任务、评测任务
6. 可观测层：日志、指标、Tracing、告警

这样回答可以把传统后端底座和 LLM 特有链路一起覆盖掉。

------

## 33. 高并发下，大模型接口的瓶颈一般在哪？

> 一般不在 Web 框架本身，而在下面几件事：

- 外部模型调用慢
- 检索和重排链路长
- 上下文拼接和序列化开销大
- 下游服务不稳定
- token 成本和配额限制

所以大模型后端优化不是只调 `worker` 数，而是要把链路拆开看。

------

## 34. 如果一个 LLM 接口很慢，你怎么排查？

我一般按链路拆：

1. 网关和应用层有没有排队
2. 缓存命中率是否下降
3. 检索、reranker、数据库分别耗时多少
4. Prompt 构造和上下文裁剪耗时多少
5. 模型首 token 延迟和总生成时长多少
6. 是否存在重试、超时、限流导致放大

核心不是猜，而是每一段都要有 trace 和 metrics。

------

## 35. Redis 在大模型系统里可以做什么？

不只是缓存，还能做：

- 会话状态缓存
- 限流计数
- 分布式锁
- 短期上下文存储
- 热门检索结果缓存
- Prompt/响应缓存
- embedding 缓存

但要注意：

> 缓存 key 必须带模型版本、Prompt 版本、检索配置，否则大模型系统很容易出现“看起来命中缓存，实际答非所问”的脏结果。

------

## 36. 向量库和关系型数据库分别负责什么？

> 关系型数据库负责结构化业务数据，比如用户、会话、权限、任务状态；向量库负责语义检索。

一般不要把二者互相替代：

- 业务过滤、事务一致性、复杂查询：关系库更强
- 语义相似度检索、Top-K 召回：向量库更强

------

## 37. RAG 系统为什么常常是“关系库 + 对象存储 + 向量库 + Redis”组合？

因为它们分工明确：

- 对象存储：原始文档
- 关系库：文档元信息、权限、任务状态
- 向量库：分块后的向量索引
- Redis：热缓存、锁、限流、短状态

真正稳定的系统，一般不是“一个数据库全干”。

------

## 38. 大模型后端要怎么做权限隔离？

至少要做三层：

- 接口层：用户身份和租户校验
- 检索层：只召回用户有权限看的文档
- 存储层：文档元数据带租户和 ACL

只在前端做权限校验没有意义，因为后端检索如果不带过滤，模型就可能看到不该看的内容。

------

## 39. 什么是上下文治理？为什么它既是 Prompt 问题，也是后端问题？

> 因为真正决定“送进模型什么内容”的，通常是后端服务，而不是前端。

常见做法有：

- 限制上下文长度
- 去重和合并相似片段
- 保留高置信和高时效信息
- 区分 system、history、retrieval context、tool result
- 对超长内容做摘要和分层裁剪

这本质上是后端在帮模型做“信息预处理”和“风险控制”。

------

## 40. 如果用户量上来了，你会先优化什么？

我通常按收益排序：

1. 缓存
2. 减少不必要的模型调用
3. 检索链路压缩
4. 批处理和异步化
5. 小模型分流
6. Prompt 与上下文瘦身

因为在大模型系统里：

> 成本优化很多时候就是性能优化。

------

## 41. 怎么做降级和兜底？

大模型系统一定要设计降级，不然下游一抖动，整条链路全挂。

常见降级策略：

- 主模型失败切备模型
- Reranker 超时则只走召回
- 工具失败时走模板化回复
- 长任务改为异步
- 流式失败降级为一次性返回

核心原则是：

> 服务可以变笨，但不能直接不可用。

------

## 42. 观测一个大模型后端，你会看哪些指标？

至少看四类：

- 稳定性：成功率、超时率、错误率
- 性能：P50/P95 延迟、首 token 延迟、队列堆积
- 质量：命中率、检索召回质量、人工抽检结果
- 成本：请求数、token 数、缓存命中率、单用户成本

如果只看接口 QPS，不看 token 和检索指标，基本管不好 LLM 系统。

------

## 43. Trace 在大模型应用里为什么比普通后端更重要？

> 因为一条请求往往会经过很多步骤：重写 query、查缓存、混合检索、重排、拼上下文、调模型、再调工具。没有 trace，你只能看到“慢”，看不到“慢在哪”。

所以我的习惯是把每一层都打 span，并把这些字段挂上去：

- 模型名
- Prompt 版本
- token 数
- Top-K
- 缓存命中情况
- 外部工具耗时

------

## 44. 如何做 Prompt / 模型 / 检索配置的版本管理？

至少要给这些东西带版本：

- Prompt 模板版本
- 模型版本
- embedding 模型版本
- 检索参数版本
- reranker 版本

否则线上效果变差时，你根本不知道是哪个改动导致的。

------

## 45. 面试官问“怎么做一个生产可用的 LLM 服务”，你怎么答？

我会按“可用、可控、可观测、可扩展”四个关键词来答：

- 可用：超时、重试、熔断、降级
- 可控：权限、限流、版本管理、配置管理
- 可观测：日志、指标、trace、告警
- 可扩展：异步任务、缓存、队列、解耦服务

如果想再升一个层次，可以补一句：

> 大模型服务比普通后端多了成本和质量两个维度，所以生产可用不仅是“服务不挂”，还包括“效果稳定、预算可控、链路可解释”。

------

## 46. 后端八股速记版

### 传统后端 10 句速记

- HTTPS = HTTP + TLS。
- TCP 重可靠，UDP 重速度。
- Cookie 在客户端，Session 在服务端，JWT 自包含。
- 幂等解决重复请求副作用。
- 限流保护稳定性和成本边界。
- Redis 不只是缓存，还常做锁、计数、会话。
- 缓存要会说穿透、击穿、雪崩。
- MySQL 索引高频考 B+ 树、回表、覆盖索引、最左前缀。
- 事务高频考 ACID、隔离级别、MVCC。
- 并发控制高频考乐观锁和悲观锁。
- 数据库扩展高频考主从复制、读写分离、分库分表。
- MQ 高价值关键词是解耦、削峰、异步、重试。

### 大模型后端 8 句速记

- SSE 很适合 token streaming。
- LLM 慢，通常慢在链路，不只慢在框架。
- 模型调用贵，所以限流、缓存、降级更重要。
- 上下文治理是后端职责，不只是 Prompt 问题。
- RAG 权限必须做在后端和检索层。
- 缓存 key 必须带模型和 Prompt 版本。
- 监控不只看 QPS，还要看 token、首 token 延迟和成本。
- 服务可以降级变笨，但不能直接崩。

### 面试收尾话术

> 我理解大模型后端和传统后端不是两套割裂的体系。传统后端负责稳定性、并发、存储和分布式基础，大模型后端是在此基础上增加模型调用、检索、上下文、成本和质量控制，所以如果底层后端能力不扎实，大模型系统也很难真正上线。
