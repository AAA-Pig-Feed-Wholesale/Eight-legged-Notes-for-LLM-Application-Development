# Agent八股

------

## 1. 什么是 LLM Agent？和普通的 RAG / Chatbot 有什么区别？

> 我理解的 Agent，其实就是一个**“会自己决策下一步干嘛的大模型应用”**。
>
> 普通 RAG 或 Chatbot，一般是固定流程：拿到问题 → 检索 → 拼上下文 → 生成，一轮就结束了。
>
> Agent 多了两件事：
>
> - 它有**状态 / 记忆**，知道之前做了什么；
>
> - 它可以**选动作**：比如先查知识库、再调工具、再调用别的模型，然后再综合这些结果回答。 所以，Agent 更像是“带大脑的工作流”，RAG 只是它的一种能力，工具调用、规划、反思都是 Agent 的范畴。

------

## 2. 一个典型的 Agent 系统一般由哪些核心模块组成？

> 我会按四块去拆：
>
> 1）**大脑（LLM）**：负责理解指令、规划步骤、决定下一步调用什么工具；
>
> 2）**工具层 / 环境**：各种 API、数据库、RAG 检索、代码执行环境等；
>
> 3）**记忆 / 状态管理**：保存历史对话、任务进度、已拿到的中间结果；
>
> 4）**调度 / 控制逻辑**：比如用 ReAct、Planner-Executor 这种模式，控制“思考 → 行动 → 观察 → 再思考”的 Loop。
>
> 我在自己的 RAG 项目里，已经有了工具层（检索、Qwen-VL）、对话记忆和缓存，下一步其实就是把控制逻辑升级成一个 Agent。

------

## 3. ReAct / Planner-Executor 这种 Agent Prompt 模式大概是怎么工作的？

> ReAct 的核心就是把模型的“思考”和“行动”显式写进 Prompt：
>
> - 先让模型用自然语言想一想当前状态、需要的信息；
>
> - 然后输出一个“Action：调用哪个工具，参数是什么”；
>
> - 系统把工具结果作为 Observation 喂回去，让模型继续下一步思考。 Planner-Executor 会更明确地分成两个模型或两个阶段：一个负责做全局规划（把任务拆成若干子任务），另一个负责执行每个子任务。 这些其实都是 Agent 的“控制套路”，在不同复杂度的场景上选不同的模式。

------

## 4. Agent 一直循环思考 + 调工具，有没有可能“跑飞”？你会怎么防止？

> 会的，典型问题就是：
>
> - 一直在“多次检索 / 多次调用 API”，但没收敛；
>
> - 或者在无意义的子问题之间来回反复。 我一般会从三层做限制： 1）**步骤上限**：设置最大回合数，比如 5 步之内必须给出结果或声明失败； 2）**工具调用白名单 + 参数检查**：不允许随便构造危险调用，对参数做基本校验； 3）**中间状态评估**：每一步都问自己“当前信息是否足够回答原问题”，足够就停止，再生成最终总结。 一旦超过上限，我会要求 Agent 给出“当前已知信息 + 为什么无法继续”的说明，而不是硬拖。

------

## 5. Agent 场景里，RAG 可以作为一种工具使用吗？你会怎么设计？

> 可以，RAG 在 Agent 体系下就是一个很典型的“查询工具”。
>
> 我的设计习惯是：
>
> - 把 RAG 封装成一个明确的 Tool：输入是结构化 query（可能包含关键词、病情摘要、图片特征），输出是文档片段列表；
>
> - 在 Prompt 里告诉 Agent：
>
> - 什么时候该用 RAG，比如“你缺乏领域知识时”；
>
> - RAG 返回的字段结构是什么；
>
> - 回答时必须引用 RAG 的证据。 这样 Agent 在规划阶段会先判断“要不要查库”，执行阶段通过 RAG 获取证据，最后再综合生成答案。

------

## 6. 多工具 Agent 里，如果有很多工具，怎么避免“工具乱调 / 调错工具”的问题？

> 工具多了以后，常见问题是：
>
> - 模型记不住每个工具的边界；
>
> - 对模糊指令会选错工具。 我的做法是两步： 1）先做一个**工具选择阶段**：可以是一个“小模型 / 分类器 / 规则”，或者让大模型先根据工具描述选择 1–2 个候选工具； 2）在 Prompt 里把每个工具的 **描述写得非常明确**：输入输出格式、典型使用场景、不要在什么情况下使用。 必要时还能加 few-shot 示例：给几个“问题→应该用哪个工具”的例子，帮模型建立映射。

------

## 7. Agent 的“记忆”都有哪些类型？你在项目里做了哪些？

> 我一般把记忆分成三层：
>
> 1）**短期记忆（对话级）**：当前会话的历史轮次、最近的操作结果；
>
> 2）**中期记忆（任务级）**：某一个任务的中间结论、已处理的子任务；
>
> 3）**长期记忆（用户 / 领域级）**：用户偏好、历史行为，或者领域知识库本身。
>
> 在医疗 RAG 项目里，我做的是：
>
> - 用 JSON 日志和 Redis 存近期对话摘要、病史、用药情况，算是短期 + 中期；
>
> - 医疗知识库本身可以理解为一种长期记忆，结合 RAG 调用。 将来做 Agent，可以让它在“每步行动前先读记忆，然后决定下一步动作”。

------

## 8. Agent 的结果怎么评估？和评估一般 RAG / Chatbot 有什么不同？

> 除了看单次回答质量，Agent 还有一个维度是**“任务完成度”和“路径质量”**。
>
> 我会关注三类指标：
>
> 1）**任务成功率**：给一批任务，看能不能在有限步数内完成，比如拿到正确信息、走对流程；
>
> 2）**行动效率**：平均调用工具次数、平均步数，越聪明的 Agent 走弯路越少；
>
> 3）**中间行为是否合理**：比如没去调不该调的高危接口，没有重复很多无用检索。
>
> 这些需要结合日志重放和标注，单纯看最后一句回答是不够的。

------

## 9. Agent 场景下，如何减少幻觉和“拍脑袋决策”？

> 减幻觉我会从两个方向入手：
>
> 1）增加**“证据驱动”的约束**：要求 Agent 在重要决策前必须先调用特定工具，比如 RAG、数据库查询，而不能凭空猜；
>
> 2）设计**“自检 / 反思”步骤**：让 Agent 在给最终答案前，检查自己的逻辑是否被返回的证据支持。
>
> 再加上一些硬性的安全规则：涉及高风险决策，必须有显式证据，否则只能给出保守结论，这样 Agent 就不敢乱拍脑袋。

------

## 10. 你觉得 Agent 和“传统编排工作流”（比如 BPMN、if-else 流程图）最大的区别是什么？

> 传统工作流是**规则写死**的：每一步走向哪个节点，基本由人编码决定，适合规则清晰、不需要深度理解的流程。
>
> Agent 则是在某些节点上，把“下一步该干嘛”的决策交给了大模型：
>
> - 它可以根据用户输入、上下文状态、工具结果，动态规划路径；
>
> - 对于非结构化输入、模糊需求更友好。 我觉得未来会是两者结合：关键路径仍然用工作流保证稳定性，中间复杂判断由 Agent 来决定具体怎么走。

------

## 11. 你有想过怎么在闻歌这种多业务场景里做 Agent 吗？

> 可以简单举个设想：
>
> - 底层有一套统一的工具层：包括内部知识库 RAG、业务 API、报表查询、搜索服务等；
>
> - 上层针对不同产品线搭不同的 Agent：比如“舆情分析 Agent”、“报告生成 Agent”、“问答客服 Agent”；
>
> - 每个 Agent 的核心差异在于 Prompt、可用工具列表和记忆策略。 我在医疗 RAG 项目里已经搭过一整套检索 + 缓存 + 上下文治理，如果有机会做闻歌的 Agent，可以把这些经验直接嵌到某个“知识咨询型 Agent”的工具层里。

------

## 12. 未来如果要把你现在的医疗 RAG 系统升级成真正的 Agent，你会做哪几步？

> 我大概会这么演进：
>
> 1）先把现有的 RAG、Qwen-VL、多轮对话记忆都封装成独立工具；
>
> 2）在 Prompt 里引入一个简单的 ReAct 模式，让模型学会“先判断任务类型，再决定是查知识库、看图片，还是直接给建议”；
>
> 3）为高风险场景增加一个“安全审查工具”，对 Agent 的草稿做二次审查和重写；
>
> 4）最后用日志记录整个“思考–行动–观察”的轨迹，针对失败案例反过来优化 Prompt 和工具设计。
>
> 这样从一个“强 RAG 系统”逐步变成一个“可规划、有记忆、有安全兜底的医疗 Agent”。

------

## 13. 单 Agent 和多 Agent（Multi-Agent）怎么选？

> 单 Agent 的优点是实现简单、状态集中、调试方便；缺点是一个模型既要规划又要执行，复杂任务下容易又慢又乱。
>
> 多 Agent 更像“角色分工”，比如一个负责规划，一个负责检索，一个负责写报告，一个负责审查。
>
> 但多 Agent 不是天然更强，它的问题也很明显：
>
> - 系统更复杂；
> - 调用次数更多；
> - 成本和延迟更高；
> - Agent 之间还会出现误传和扯皮。
>
> 所以我的原则是：任务边界清晰、单 Agent 扛不住时，才考虑多 Agent，而不是为了“听起来高级”硬拆。

------

## 14. 为什么 Agent 场景里特别强调“结构化动作”和 schema？

> 因为如果让 Agent 自由输出一大段自然语言，再由系统去猜它想干嘛，错误率会非常高。
>
> 更稳的做法是让 Agent 输出结构化动作，比如：
>
> - 工具名
> - 参数
> - 置信度
> - 是否继续
>
> 这样系统更容易校验、回放和审计。
>
> 这也是为什么现在很多 Agent 系统会把“思考过程”和“可执行动作”分开，前者可以半自由，后者尽量结构化。

------

## 15. MCP 是什么？为什么它在 Agent 场景里越来越常见？

> MCP，全称 Model Context Protocol，可以把它理解成“模型接工具和数据源的通用接口标准”。Anthropic 官方把它类比成 AI 应用的 USB-C，OpenAI 也已经支持 MCP 相关接入。  
> 来源：<https://docs.anthropic.com/en/docs/mcp> 、<https://platform.openai.com/docs/mcp/>
>
> 它的价值是：
>
> - 不用每接一个工具都重新造一套协议；
> - 更方便把搜索、数据库、知识库、业务系统统一暴露给 Agent；
> - 更利于生态复用。
>
> 面试里一句话就够：
>
> > MCP 不是某个具体 Agent 框架，而是让 Agent 更容易接外部能力的标准化协议。

------

## 16. Agent 为什么常常需要 Human-in-the-Loop？

> 因为很多高风险操作不能完全放给模型自己做，比如：
>
> - 发邮件
> - 执行支付
> - 改数据库
> - 医疗或合规建议
>
> Human-in-the-Loop 的核心不是“人来替模型思考”，而是在人机边界上加审批点：
>
> - 高风险动作前要人工确认；
> - 低风险动作允许自动执行；
> - 全链路可回放、可追责。
>
> 这会让 Agent 更像真实生产系统，而不是实验室 Demo。

------

## 17. 什么时候不该用 Agent？

> 这是面试里很加分的问题。
>
> 我觉得下面几种情况不该上 Agent：
>
> - 流程本来就很固定，if-else 就能解决；
> - 任务不需要多步决策；
> - 对延迟和成本极其敏感；
> - 高风险动作很多，但又没有审查和回滚机制。
>
> 换句话说，Agent 适合“路径不确定但目标明确”的问题，不适合“规则已经写死”的问题。

------

## 18. Agent 系统怎么做更完整的评估？

> 除了看最终答得对不对，我还会看四类指标：
>
> - 任务成功率
> - 平均步数和平均工具调用次数
> - 单次任务成本和时延
> - 风险动作触发率、错误工具调用率
>
> 如果要更进一步，我会保留轨迹日志，然后做人类审查：
>
> - 规划是否合理
> - 有没有无效反复
> - 有没有跳过必要证据
>
> 这类评估比普通 Chatbot 更像“过程评估”，不只是结果评估。
