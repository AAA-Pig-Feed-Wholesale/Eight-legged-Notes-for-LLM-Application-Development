# Python八股

------

## 1. 为什么大模型应用开发岗位也会问 Python？

> 因为大模型应用开发不是只会调 API。真正落地时，数据清洗、服务开发、异步并发、任务编排、缓存、日志、评测脚本，很多都落在 Python 上。面试官问 Python，本质上是在看你能不能把模型能力变成稳定可运行的系统。

------

## 2. `list`、`tuple`、`set`、`dict` 的区别？

> `list` 有序可变，适合顺序数据；`tuple` 有序不可变，适合配置和常量；`set` 无序去重，适合判重和集合运算；`dict` 是键值映射，适合结构化数据和索引。
>
> 面试里我一般会补一句：大模型应用里，`dict` 很常用于请求参数、Prompt 模板变量、检索结果元数据；`set` 常用于去重 doc_id 或已访问节点。

------

## 3. 浅拷贝和深拷贝有什么区别？

> 浅拷贝只复制最外层对象，内部嵌套对象还是共享引用；深拷贝会递归复制，内部对象也独立。
>
> 这个问题在 Prompt 模板、配置字典、Agent 状态上下文里特别容易踩坑。如果你只是 `copy.copy()` 一个嵌套字典，再改内部字段，很可能把原请求上下文也改掉。

------

## 4. 为什么不要把可变对象写成默认参数？

> 因为函数默认参数在定义时只初始化一次，不是每次调用都新建。像 `def f(x, cache=[])`，这个 `cache` 会在多次调用之间共享。
>
> 正确写法一般是：
>
> ```python
> def f(x, cache=None):
>     if cache is None:
>         cache = []
> ```
>
> 这是 Python 面试高频坑，也是线上状态污染的真实来源。

------

## 5. `==` 和 `is` 有什么区别？

> `==` 比较值是否相等，`is` 比较是否是同一个对象。
>
> 所以判断 `None` 要用 `is None`，不要用 `== None`。这是代码规范问题，也是避免对象重载比较运算导致歧义的问题。

------

## 6. 迭代器、生成器、可迭代对象分别是什么？

> 可迭代对象是实现了 `__iter__()` 的对象，比如 `list`、`dict`；迭代器是可以不断调用 `next()` 取值的对象；生成器本质上是一种特殊迭代器，通常由带 `yield` 的函数生成。
>
> 在大模型开发里，生成器很常见，比如流式返回 token、分批处理文档、逐段读取大文件，都适合用生成器降低内存占用。

------

## 7. 装饰器是干什么的？你在项目里会怎么用？

> 装饰器本质上是“在不改原函数代码的前提下，为函数增加额外逻辑”的语法糖。
>
> 项目里常见用途包括：
> - 接口耗时统计
> - 重试与熔断
> - 权限校验
> - 日志埋点
> - 缓存
>
> 如果面试官继续问，我会强调：装饰器适合横切逻辑，但别把核心业务全塞进去，否则调试很痛苦。

------

## 8. `*args` 和 `**kwargs` 有什么用？

> `*args` 接收位置参数，`**kwargs` 接收关键字参数。它们常用于封装通用组件，比如统一的模型调用器、重试器、日志包装器。
>
> 在 LLM 项目里很常见，因为不同模型提供商的参数不完全一样，`**kwargs` 适合做透传。

------

## 9. 什么是上下文管理器？为什么 `with` 很重要？

> 上下文管理器通过 `__enter__` 和 `__exit__` 控制资源的申请与释放，`with` 能保证文件句柄、数据库连接、锁等资源在异常场景下也被正确释放。
>
> 比如读文件、写日志、加锁、开启 trace span，都适合 `with`。这体现的是工程稳定性。

------

## 10. Python 的异常处理应该怎么设计？

> 我的习惯是三层：
> - 底层保留原始异常，别吞异常
> - 中间层做语义化包装，比如 `ModelTimeoutError`
> - 接口层统一转成可观测、可返回的错误码和日志
>
> 面试里不要只说 `try/except/finally`，更重要的是体现你知道“错误要可定位、可恢复、可监控”。

------

## 11. 线程、进程、协程有什么区别？

> 线程适合 I/O 并发，进程适合 CPU 密集型任务，协程适合高并发 I/O 且切换成本低。
>
> 在大模型应用开发里：
> - 调外部 LLM API、查数据库、调向量库，通常偏 I/O，适合异步或线程池
> - 文档切分、批量 embedding、离线评测，有些偏 CPU，可能更适合多进程
> - 如果涉及本地推理，还要考虑模型显存和进程复制成本

------

## 12. 什么是 GIL？它会带来什么影响？

> GIL 是 CPython 的全局解释器锁，同一时刻只有一个线程执行 Python 字节码。所以 Python 多线程对 CPU 密集型任务提速有限。
>
> 但这不代表线程没用。I/O 阻塞时线程会释放 GIL，所以网络请求、数据库访问、文件读写这类任务，多线程仍然有效。

------

## 13. `async/await` 的本质是什么？

> 本质是协程和事件循环。`await` 表示当前协程在等待一个异步结果，控制权交回事件循环，让它去执行别的任务。
>
> 适合大量 I/O 并发场景，比如：
> - 并发请求多个模型或工具
> - 并发查检索服务、重排服务、缓存
> - SSE 流式输出
>
> 但如果你在 `async def` 里直接写阻塞代码，比如同步请求或大规模 CPU 计算，事件循环一样会被卡死。

------

## 14. FastAPI 里 `async def` 就一定更快吗？

> 不一定。只有当内部逻辑真的是异步 I/O 时，`async def` 才更有价值。如果里面还是同步数据库驱动、同步 HTTP 请求、重 CPU 计算，那只是“异步外壳套同步内核”。
>
> 面试里这样答会更工程化：`async` 不是银弹，关键是整条调用链是不是异步。

------

## 15. Python 里常见的性能优化手段有哪些？

> 我一般分四类讲：
> - 少拷贝：避免重复序列化、重复拼接大字符串
> - 少阻塞：I/O 用异步、线程池或连接池
> - 少重复：用缓存、批处理、向量化
> - 少热点落在 Python 解释层：必要时用 C 扩展库、NumPy、Rust 扩展或把重任务下沉到服务层
>
> 在 LLM 系统里，性能瓶颈很多时候不在 Python 语法本身，而在网络、模型调用、序列化和上下文拼接。

------

## 16. 多进程加载大模型要注意什么？

> 核心问题是内存和显存复制。
>
> 如果是本地模型服务，多进程 worker 不一定更好，因为每个进程都可能复制一份模型权重，导致内存爆炸。通常要结合部署方式判断：
> - 远程模型 API：应用层可以多 worker
> - 本地单卡模型：更偏向单模型服务 + 外部网关
> - 多卡推理：交给专门推理框架，而不是纯 Python 硬顶

------

## 17. `dataclass`、`pydantic`、普通类分别适合什么场景？

> `dataclass` 适合轻量数据对象；`pydantic` 适合接口入参、配置校验和序列化；普通类适合复杂业务逻辑。
>
> 做 LLM 应用后端时，我更常把：
> - 请求响应模型交给 `pydantic`
> - 内部状态对象用 `dataclass`
> - 真正的服务对象写成普通类

------

## 18. 日志在 Python 项目里应该怎么打？

> 不要只会 `print`。正式项目至少要做到：
> - 用结构化日志
> - 带 request_id / trace_id
> - 关键字段打出来，比如模型名、token 数、延迟、命中缓存与否
> - 错误日志带堆栈
>
> 对 LLM 项目来说，日志不是可选项，因为没有日志你根本没法排查幻觉、超时、成本飙升和 Agent 跑飞。

------

## 19. 面试里怎么把 Python 回答得更像“大模型应用开发”？

> 不要停留在语法定义，要主动把 Python 和实际场景绑定：
>
> - 生成器 → 流式输出 / 大文件处理
> - 异步 → 并发调模型 / 检索 / 工具
> - 装饰器 → 埋点 / 重试 / 缓存
> - 上下文管理器 → 连接、锁、trace 管理
> - GIL → 为什么 I/O 用线程/协程，CPU 用多进程
>
> 这样面试官会觉得你不是在背 Python 书，而是在解决真实系统问题。

------

## 20. Python八股速记版

### 一句话记忆

- 可变默认参数是坑，用 `None` 兜底。
- `==` 比值，`is` 比对象身份。
- 生成器适合流式和省内存。
- `async` 适合 I/O，不适合假异步。
- GIL 限制 CPU 多线程，不妨碍 I/O 并发。
- 装饰器管横切逻辑，别滥用。
- `with` 的价值是资源安全释放。
- 正式项目靠结构化日志，不靠 `print`。

### 面试收尾话术

> 我理解 Python 在大模型应用岗位里不是考语法细节本身，而是考你能不能把异步、并发、异常、日志和数据结构这些基础能力，真正用到模型服务、检索链路和 Agent 编排里。
