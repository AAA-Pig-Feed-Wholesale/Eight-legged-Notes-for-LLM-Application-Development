# RAG八股

------

### Q1：什么是 RAG？相比直接用大模型有什么优势？

> 我自己理解的 RAG，就是 **Retrieval-Augmented Generation**，检索增强生成。
>
> 普通大模型是：输入问题 → 模型凭记忆直接生成；
>
> RAG 是：
>
> 1）先把业务知识做成向量库 / 索引；
>
> 2）用户提问后先检索相关文档片段；
>
> 3）再把问题 + 检索到的证据一起喂给大模型生成。
>
> 好处有三点：
>
> - 知识更新只需要重建索引，不用重训模型；
>
> - 回答能绑定具体文档，有溯源，易审计；
>
> - 幻觉会明显减少，因为模型是“看着材料写作文”，而不是自己瞎编。

------

### Q2：典型的 RAG 流程分哪几步？每一步可能踩的坑是什么？

> 我一般把 RAG 流程拆成 5 步：
>
> 1）数据清洗和分块：容易踩坑在“块太碎/太长”导致信息丢失或噪声多；
>
> 2）向量化 & 建索引：embedding 不匹配领域，召回就会很迷；
>
> 3）检索（向量 / BM25 / 混合）：只用一种检索，容易错过关键文档；
>
> 4）上下文治理（Top-K 选择、去重、摘要）：K 乱设，模型不是看不懂，就是被信息淹没；
>
> 5）生成 & 后处理：prompt 不规范和安全层缺失，很容易幻觉。
>
> 我的习惯是先搭一个简单 baseline，把每一步指标和日志打通，再有的放矢地优化。

------

### Q3：RAG 和微调各自适合什么问题？如果一个业务问答系统让你选，你会怎么搭？

> 微调本质是把知识和行为“写进模型参数”；RAG 是把知识放在外面，让模型去查。
>
> 如果业务知识更新频繁、需要引用具体条款，比如医疗、金融、政务，我会优先上 RAG；
>
> 如果是偏 **打分、排序、分类、风格对齐**，比如你的偏好预测、质量打分，我会用微调模型。
>
> 实际项目里可以结合：RAG 负责找证据，微调模型负责打分和排序，这样既有溯源又有“智能评分”。

------

### Q4：RAG 里的分块（chunk）怎么设计？为什么？

> 我会先看业务文档的结构。
>
> 医疗说明书这类文档通常有“适应症、用法用量、禁忌、注意事项”等结构，所以我不会死按 256/512 token 硬切，而是：
>
> - 先按章节/小标题拆，再在每节内部用 512 左右长度滑动分块，加一点重叠； 好处是：
>
> - chunk 自己是自洽的小段话，不容易语义断裂；
>
> - 又不会太长，把一堆完全不相关的信息绑在一起。 实践里可以用 ragas 或人工抽样比较不同 chunk 策略下的 recall/precision，再确定最终配置。

------

### Q5：RAG 里如何选择 embedding 模型？你自己是怎么做的？

> 我会按 **语言 + 领域** 两个维度来选。
>
> 一开始可以用 all-MiniLM 这类通用英文模型先跑通流程；
>
> 但在中文/中医这种混合语料上，我会更偏向 **专门做中文 / 多语种的 embedding**，比如 Qwen-3 text-embedding-v3。
>
> 真正怎么选，我的做法是：
>
> - 先用简单验证集 + ragas 比较不同 embedding 下的 context_recall/precision；
>
> - 再做一些典型 query 的人工抽样。 在医疗 RAG 项目里就是这样：all-MiniLM 做 baseline，然后用 Qwen-3 提升了中医/西医混合语料的召回效果。

------

### Q6：为什么要做 BM25 + 向量混合检索？只用向量不行吗？

> 只用向量其实能跑，但在一些靠 **“关键字精确匹配”** 的场景会比较危险，比如药名、病名、法律条款号。
>
> 所以我更喜欢 BM25 + 向量混合：
>
> - BM25 兜底关键词匹配，保证“叫这个名字的文档”一定在候选里；
>
> - 向量检索补充语义相近、表达不完全一样的内容；
>
> - 最后用 MMR 或线性融合 + reranker 排个序。 实际测试下来，混合检索在 recall 和 precision 上都比单一向量要稳定，尤其是短 query、专业术语很多的场景。

------

### Q7：为什么在 RAG 里要用 reranker？你会怎么选 / 怎么用？

> 第一层检索通常会召回一个 Top-K 列表，但这里面难免有“看起来有点相关但不是最核心”的文段。
>
> reranker 的作用就是在 Top-K 里 **再做一轮精排**：
>
> - 输入是 query + 文档对，输出是相关性分数；
>
> - 训练时会用对比学习，让分数能区分“真相关”和“聊胜于无”。 我会选一个对中文友好的模型，比如 bge-reranker-v2-m3，或者 Qwen 系列的 reranker，在 K 比较小（比如 20 左右）时 rerank 一下，再把排在前几的文档喂给大模型。 这样可以显著提升 context_precision，减少“答非所问”的情况。

------

### Q8：如何评估一个 RAG 系统好不好？你会看哪些指标？

> 我会分三层看：
>
> 1）**检索层**：用 ragas 看 context_recall、context_precision、平均检索时延；
>
> 2）**回答层**：在一小部分验证集上做人评，看“是否答到点上、有没有危险建议”；
>
> 3）**系统层**：看端到端响应时间、token 成本、用户反馈。
>
> 现在项目里基本是 ragas + 人工抽样结合，先确保检索质量和时延在一个合理区间，再去调生成的行为和安全性。

------

### Q9：RAG 的 Top-K 怎么选？K 大一点是不是一定更好？

> Top-K 是个典型的 trade-off：
>
> - K 太小，容易漏掉关键证据，recall 不够；
>
> - K 太大，precision 会下降，模型上下文被一堆噪声占满，成本也会明显上去。 我的做法是用 ragas 做一个小实验，比如 K=3、5、8，分别看 recall、precision 和时延。 像我医疗 RAG 项目里，最后选的是 Qwen-3 + FAISS + K=3，这个配置下 recall 能到 0.81 左右，precision 也在 0.78，整体成本可控，比一味堆 K 更划算。

------

### Q10：RAG 里经常说“上下文治理”，具体你做了什么？

> 上下文治理我一般拆成三件事：**选择、压缩、缓存**。
>
> - 选择：只保留 Top-K 高相关证据，历史对话只保留最近几轮的关键内容；
>
> - 压缩：对超长的历史对话和文档，先做结构化摘要或裁剪，只保留和当前问题强相关的要点；
>
> - 缓存：把检索结果和最终回答按“问题 + 图像哈希 + 关键病史”等做缓存，重复问题直接复用。 这样可以在保证回答质量的前提下，控制上下文长度和 token 成本，尤其适合医疗问诊这种多轮场景。

------

### Q11：RAG 实际应用中常见的错误类型有哪些？你是怎么 debug 的？

> 我遇到的主要有三类：
>
> 1）**召回不全**：用户问的点知识库里有，但没被检索到，多半是 embedding / 分块 / K 设置问题；
>
> 2）**召回乱**：检索到了很多看起来相关、其实不太有用的段落，通常是 recall 高 precision 低，需要调权重或加 reranker；
>
> 3）**回答乱编**：有证据但模型没好好用，或者证据确实不足，这就要从 prompt、安全规则和高风险过滤上改。
>
> debug 的时候，我会把 query、Top-K 文档和回答一起打印出来，看是“找不到、找不对，还是用不好”，然后针对那一层去调。

------

### Q12：RAG 知识库怎么更新？要不要每次都重新建库？

> 跟直接微调相比，RAG 更新知识库会简单很多。
>
> 一般做法是：
>
> - 新文档来了之后，做同样的数据清洗、分块和 embedding；
>
> - 把新的向量追加写入向量库，或者做增量索引；
>
> - 旧版本的文档可以带上 version、时间戳，必要时做失效标记。 不需要每次重建整个库，更不需要重训大模型。 真正成本在于构建一个自动化的“抓取→清洗→分块→索引”的 pipeline，而不是模型本身。

------

### Q13：RAG 如何配合多模态，比如图片、表格？

> 思路是一样的：先把图片/表格变成**可索引的文本或结构化特征**，再走一遍 RAG。
>
> 比如在医疗项目里，我用 Qwen-VL 把舌象/处方图片解析成“舌色、苔色、药名、剂量”等信息，再把这些信息拼进检索 query 和 prompt 里。
>
> 这样图片不是简单当背景图，而是直接变成检索信号和生成条件。
>
> 后面也可以考虑给图片单独建一个“多模态索引”，文本和图片两个通道的检索结果融合起来用。

------

### Q14：如果让你用 RAG 给一个全新垂直领域（比如法律/金融）做问答，你会怎么落地？

> 我会按这么几个步骤来：
>
> 1）先和业务确认“问题范围”和“优先支持的文档类型”；
>
> 2）搭一版最小可用的 RAG：选一个合适语种的 embedding，做基础向量检索；
>
> 3）基于这版 baseline，收集真实 query，构建一个小的验证集；
>
> 4）再逐步加混合检索、reranker、上下文治理和日志评估。
>
> 同时会把业务方拉进来做一批人评，尤其是高风险问题，这些数据未来还可以用来做 SFT 或 DPO，对齐行为。

------

### Q15：长上下文模型越来越强了，为什么还要做 RAG？

> 长上下文确实能减少一部分检索误差，但它不等于 RAG。
>
> RAG 仍然有几个核心价值：
>
> - 知识库可以持续更新；
> - 可以做权限过滤；
> - 可以给出引用和溯源；
> - 在大知识库场景下，通常比“整库硬塞”更省成本。
>
> 所以我更倾向把长上下文看成 RAG 的增强项，而不是替代品。

------

### Q16：RAG 里的 metadata filtering 和 ACL 为什么重要？

> 真正线上 RAG，不只是“找相关”，还要“找用户有权限看的相关”。
>
> 所以检索时除了语义相似度，我还会结合 metadata 过滤，比如：
>
> - 租户 ID
> - 文档类型
> - 时间范围
> - 权限标签
>
> 否则即使召回很准，也可能把不该看的文档喂给模型，这是生产事故。

------

### Q17：什么是 Self-RAG / selective retrieval？它和普通 RAG 有什么区别？

> 普通 RAG 往往是“无论问题需不需要，都先检索固定 Top-K”；Self-RAG 这类方法更强调“按需检索”和“自我反思”。
>
> Self-RAG 论文的核心思想就是：模型自己判断什么时候该检索、怎么利用检索结果，并对生成内容做反思控制。  
> 来源：<https://arxiv.org/abs/2310.11511>
>
> 面试里不用讲太细公式，讲清楚趋势就够：
>
> > RAG 正在从“固定流程检索”往“按需检索、会反思、会筛证据”的方向发展。

------

### Q18：GraphRAG 是什么？什么时候值得考虑？

> GraphRAG 可以理解成“把文档不只是切块存向量，还进一步抽成图结构”，然后利用实体、关系、社区摘要去支持更复杂的检索和推理。
>
> Microsoft Research 2024 年把 GraphRAG 开源后，这个方向关注度很高，特别适合：
>
> - 多跳推理
> - 关系密集型知识
> - 大量跨文档关联问题  
> 来源：<https://www.microsoft.com/en-us/research/blog/graphrag-new-tool-for-complex-data-discovery-now-on-github/>
>
> 但它也更重，图构建和维护成本更高，所以不是所有业务都值得一上来就做 GraphRAG。

------

### Q19：怎样让 RAG 的回答更“有依据”，而不是只把文档塞进去？

> 我会做三件事：
>
> - 在 prompt 里要求回答必须绑定证据；
> - 检索结果里保留文档 ID、标题、段落来源；
> - 对证据不足的情况允许 abstain，也就是明确说“不确定”。
>
> 这样做的目标不是让模型显得更会说，而是让它在“知道”和“不知道”之间有清晰边界。

------

### Q20：RAG 未来的优化方向，你会关注哪些？

> 如果从近两年趋势看，我会重点关注四类：
>
> - 更聪明的 query rewriting 和 query decomposition
> - selective retrieval / Self-RAG 这类按需检索
> - GraphRAG 这类结构化检索
> - 更细粒度的 RAG 评测和可解释诊断
>
> 换句话说，RAG 的演进方向不是“再堆一个向量库”，而是让检索、更少噪声和生成之间的协同更强。

### Q1：稀疏检索（BM25）和稠密检索（向量检索）有什么区别？

> 稀疏检索就是传统倒排索引，比如 BM25，它靠**关键词精确匹配**，对专业名词、编号类问题特别靠谱，但表达稍有变化就容易漏；
>
> 稠密检索是用 embedding 把文本映射到向量空间，靠**语义相似度**来找文档，对“同义改写、说法多样”的问题更友好；
>
> RAG 里面我一般是两者结合：BM25 兜底关键字，向量补充语义相近的内容，再用 reranker 做精排。

------

### Q2：为什么向量检索通常要“分块”到段落级，而不是直接对整篇文档做 embedding？

> 如果只对整篇文档做 embedding，检索时虽然能找到对的文档，但返回的上下文会非常长，和问题相关的内容只是其中几句话，既浪费 token，又增加噪声；
>
> 按段落或小节分块后，每个块只聚焦一个小主题，检索到的内容更精确，模型也更容易利用；
>
> 所以我会先按章节或逻辑结构分段，再按长度做二次分块，保证每个 chunk 自洽又不过长。

------

### Q3：什么是向量库 / 向量索引？为什么要用 FAISS 之类的库？

> 向量库就是存放“文本 embedding”并支持**相似度检索**的组件；
>
> 小规模数据可以暴力算相似度，但一旦上到几十万、几百万条，就需要像 FAISS 这种**近似最近邻索引**，用专门的数据结构（比如 IVF、HNSW）来加速检索；
>
> 简单说，就是用 FAISS 这种库，能在可接受的时间内从海量向量里找出 Top-K 最相似的几条。

------

### Q4：什么是 MMR（Maximal Marginal Relevance）？在 RAG 里有什么用？

> MMR 的核心就是两个字：**多样性**。
>
> 普通 Top-K 只看“和 query 多像”，可能会给出很多内容几乎重复的文段；
>
> MMR 在打分时既考虑“和 query 的相似度”，也考虑“和已选文档的差异度”，优先选**既相关又不太重复**的内容；
>
> 在 RAG 里用 MMR 能减少上下文里的“同义堆砌”，提高信息覆盖率。

------

### Q5：RAG 里的 Query Rewriting 是什么？什么时候会用？

> Query Rewriting 就是先把用户的原始问题**改写成更适合检索的版本**，再去查库；
>
> 比如对话上下文很长、用户只说“那这种情况呢？”这类省略问句时，直接检索会完全找不到信息；
>
> 我会让大模型先“补全上下文 +抽取关键要点”，生成一个独立的问题，比如“对于 65 岁高血压患者服用 XX 药物时，需要注意哪些不良反应？”，再用这个问题去做检索，召回会稳定很多。

------

### Q6：Cross-Encoder 和 Bi-Encoder 在检索 / 重排里有什么区别？

> Bi-Encoder 是“各自编码”：query 编一次、文档编码一次，检索时只要算向量相似度，速度快，适合做第一层召回；
>
> Cross-Encoder 是把 query 和文档拼在一起，让模型一起看，然后直接输出一个相关性分数，效果通常更好，但是要**每对都前向一次，开销很大**，所以多用在 Top-K 的 rerank 阶段；
>
> 我在项目里就是用 Bi-Encoder 做向量检索，再用像 bge-reranker 这种 Cross-Encoder 风格的模型做精排。

------

### Q7：RAG 如何控制延迟？如果用户量上来了，你会怎么优化？

> 延迟主要花在两块：检索和大模型生成。
>
> 检索侧可以做的包括：
>
> - 用合适的索引结构（比如 FAISS IVF/HNSW），控制向量维度和 Top-K；
>
> - 做缓存，对高频问题和检索结果做缓存； 模型侧可以：
>
> - 控制输入上下文长度，只保留最有用的证据；
>
> - 对于固定格式的回答，用更小的模型或工具链处理一部分逻辑。 我自己在医疗 RAG 项目里就通过 K 调参和缓存，把平均检索时延压到了几秒以内。

------

### Q8：RAG 能完全消除大模型的幻觉吗？你是怎么降低幻觉的？

> 说实话，RAG **不能 100% 消除幻觉**，但可以大幅降低；
>
> 我的做法是几层：
>
> 1）检索层尽量保证召回质量，比如合理分块、混合检索、reranker；
>
> 2）Prompt 层要求模型“只能基于提供的证据回答，不要编造额外事实”；
>
> 3）回答层做一些检查，比如让模型显式列出“依据片段”，或者用额外的判别模型做事实一致性校验；
>
> 对高风险场景，证据不足时我会要求模型必须说“不确定/建议线下就诊”，而不是硬编。

------

### Q9：RAG 在多源知识（结构化表 + 文档 + 日志）场景下怎么做？

> 我会把不同类型的知识分成不同“通道”：
>
> - 文本类走常规向量检索；
>
> - 结构化表走数据库查询或 NL2SQL；
>
> - 日志类可能做成时间序列或事件索引； Agent 或调用层负责根据问题类型选用哪几个通道，然后把不同来源的结果统一成“证据块”形式，再交给大模型生成。 这样既能利用 RAG 的优势，又不会把所有东西硬塞进一个向量库里。

### Q10：如果未来要在 RAG 上做“学习型检索”（自己训练一个 retriever），大概思路是什么？

> 现在大多数 RAG 用的是现成的 embedding 模型，其实也可以在行业数据上再训一个 retriever；
>
> 大致思路是：
>
> - 构造一批“query–document”对，正样本是真正相关的片段，负样本是随机/困难样本；
>
> - 用对比学习（contrastive learning）去训练一个 Bi-Encoder，让正样本距离更近、负样本更远；
>
> - 训练好的 encoder 就可以替换现有 embedding 模型继续用。 这个工作量会比直接用通用 embedding 大很多，但在垂直领域（比如医疗、法律）理论上能换来更高的检索质量。

### 什么是召回率

召回率这个东西，其实就是在问一句话：

> **“所有真正相关/真正为正的样本里，有多少被我找出来/判对了？”**

------

1. 形式一点的定义

- 假设一共真的有 **N 个“该被找出来的东西”**（真正正样本 / 相关文档）。
- 你的模型只找出了其中的 **M 个**。

那 **召回率（Recall） = M / N**。

公式写就是：

[

\text{Recall} = \frac{\text{True Positive}}{\text{True Positive}+\text{False Negative}}

]

- True Positive（TP）：该找出来而且你找出来了
- False Negative（FN）：该找出来但你漏掉了

------

2. 一个小例子（RAG 视角）

你有一个知识库，里面有 10 段和“阿司匹林禁忌”相关的内容。

用户问了一个关于禁忌的问题，理想情况下这 10 段都算“真正相关”。

- 检索返回了 7 段是真正相关的，另外 3 段没召回。
- 那这一问的召回率 = 7 / 10 = 0.7。

**含义：**

> 该给模型看的“关键信息”有 10 段，我只给到了 7 段，还有 3 段漏掉了。

------

3. 和精准率的对比（顺便）

- **召回率高**：漏的少，但可能捞了一堆不太相关的东西一起上来；
- **精准率（Precision）高**：捞上来的大部分是对的，但可能漏掉不少该捞的。

RAG 里经常就是在“召回率”和“精准率”之间做权衡，比如 Top-K 取 3 还是 8。



### 重排是干什么？

很好，正好是 RAG 里的关键概念。

**一句话：**

> 重排（rerank）就是在**已经召回的一小撮候选里，再排一遍名次，挑出“最相关、最有用的前几条”**。

------

为什么要重排？

第一步召回（BM25 / 向量检索）通常只是“**快速大致找一批可能相关的文档**”，比如先从十万条里捞出 50 条。

但这 50 条里：

- 有些跟问题非常贴；
- 有些只是蹭了几个关键词，其实帮助不大。

这时候就需要一个更聪明、但稍微贵一点的模型来**重新打分排序**——这一步就是 **rerank**。

------

它具体在干什么？

流程可以这样理解：

1. 粗召回

   ：

   - 用 BM25 / 向量检索，从大量文档里选出 Top-50；

2. 重排

   ：

   - 把“问题 + 每一条候选文档”成对丢给一个更强的模型（比如 bge-reranker、cross-encoder），
   - 这个模型给出一个相关性分数；
   - 按这个分数重新排序，选出 Top-3/Top-5 作为最终喂给大模型的上下文。

3. 所以 **重排的重点是“在小范围内提高排序质量”**，不负责从海量库里捞东西，那是召回的工作。

4. 

5. 用你项目的话来翻译：

6. 你可以在面试里这么解释自己做的事：

7. 在我的 RAG 项目里，第一步是用向量检索/混合检索把候选文档召回出来，比如 Top-50。

   但是这 50 条里相关性参差不齐，所以我再用 Qwen2.5-reranker / bge-reranker 这类 cross-encoder 做了一步 **rerank**，让模型同时看“问题+文段”，输出一个相关性分数，最后只保留前 3 条或前 5 条给大模型。

   这样可以明显提升最终回答的相关性和稳定性。

9. embedding的基本原理

10. 我给你一个**面试版本**的回答思路，先是一段可以直接说的，后面再补一点细节。

11. 

12. 1. 可以直接说出口的版本（30–40 秒）

13. 我理解的 embedding，本质上就是一个**把离散的文本映射成连续向量空间的函数**。

    模型会学到一个映射 (f(\text{text}) \rightarrow \mathbb{R}^d)，让**语义相近的句子 / 词在向量空间里离得更近，语义无关的离得远**。

    训练的时候通常用对比学习或语言模型的信号：

    - 对“正样本对”（比如同一问答对、相似句）拉近向量距离；

    - 对“负样本对”拉远距离。 最后我们就可以用点积 / 余弦相似度在这个空间里做相似度检索，比如在 RAG 里用 embedding 找最相关的文档片段。

14. 

15. 2. 稍微展开一点（如果面试官追问）

16. （1）为什么要 embedding？

17. 

18. 原始文本是离散符号（字、词），模型没法直接“算距离”；

19. 先前的 one-hot 表示维度巨大、全是 0，也表达不了语义关系；

20. 所以用一个神经网络，把文本编码成

    定长、稠密的实数向量

    ，让“语义关系 → 几何关系”：

    - 相似 → 向量距离小；
    - 不相似 → 距离大。

21. （2）一个典型 sentence embedding 的流程

22. 以“句向量 embedding 模型”为例（比如你项目里用的 Qwen-3 / BGE 系列）：

23. 

24. **分词 / 分字**：把句子切成 token；

25. **编码**：通过 Transformer 等编码器，把每个 token 变成一个向量（hidden states）；

26. 池化（pooling）

    ：把一整句的 token 向量压成一个句向量，例如：

    - 取 [CLS] 位置；
    - 或对所有 token 做平均 / 加权平均；

27. **归一化**：经常会做 L2 Norm，方便后面用余弦相似度。

28. 得到的就是一个维度固定的向量，比如 768 维、1024 维。

29. （3）怎么训练出“语义相近靠得近”的性质？

30. 常见两种思路：

31. 

32. 语言模型信号 + 池化

    （BERT 类）

    - 用 MLM / 下一句预测等任务预训练，让编码器学到语义表示；
    - 再对整句做 pooling，当作 sentence embedding。

33. 对比学习 / 双塔（现在很常用）

    - 准备很多“query–positive–negative”三元组：
      - 正样本：真正相关 / 同义表达；
      - 负样本：随机句子或难负样本；
    - 编码后让模型最小化： [ \text{sim}(q, p) \gg \text{sim}(q, n) ]
    - 这样训练多了，自然就学会：相似的句子向量靠近，不相似的拉远。

34. （4）在 RAG / 检索里怎么用？

35. - 离线阶段：
      - 对每个文档 chunk 做 embedding，存到向量库；
    - 在线检索：
      - 对用户 query 做同一个模型的 embedding；
      - 在向量空间里用余弦 / 点积找 Top-K 最近的文档；
      - 把这些文档连同问题一起喂给大模型生成答案。

36. 你可以一句话收尾：

37. > 所以 embedding 的基本原理就是：**用神经网络把文本压进一个语义空间里，让“语义相似 → 向量相似”，方便后续做相似度检索和下游任务。**
